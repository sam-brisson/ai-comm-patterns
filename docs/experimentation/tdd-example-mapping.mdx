# TDD with Example Mapping

How does a rule discovered in a workshop become working, tested software? This interactive guide walks through the connection between Example Mapping and Test-Driven Development.

<TDDKnowledgePage />

## The Connection

The examples you write during Example Mapping **are** your test cases. Each green card ("Both empty → disabled") becomes a test that the engineer writes before implementing the feature.

This is the heart of TDD:
1. **Write the test first** (from your example)
2. **Watch it fail** (the bug exists)
3. **Write just enough code** to make it pass
4. **Refactor** if needed (tests ensure nothing breaks)

## Why This Matters

For **Product Managers**: Your examples directly shape the tests. The workshop isn't just requirements gathering — it's writing tests in plain English.

For **Engineers**: Example Mapping gives you concrete test cases before you write a line of code. No guessing what to test.

For **Teams**: A shared understanding of what "done" looks like, captured as both human-readable examples and executable tests.
